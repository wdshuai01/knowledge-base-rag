# 研发发布与故障处理制度

## 1. 版本计划

研发团队按双周迭代节奏制定版本计划。每个版本应明确需求范围、负责人、测试计划、发布时间和回滚方案。

需求进入开发前，产品经理应完成需求说明，研发负责人应完成技术评审。未经评审的需求不得直接进入开发排期。

## 2. 代码管理

所有代码必须提交到公司代码仓库。禁止将公司源代码保存到个人网盘、个人 Git 仓库或未经批准的外部平台。

主分支受到保护，开发人员应通过合并请求提交代码。每个合并请求至少需要一名同组研发人员 Review。涉及支付、权限、数据删除和安全策略的代码，需要研发负责人额外审核。

提交信息应描述变更目的和影响范围，不得使用无意义描述。

## 3. 测试要求

上线前必须完成单元测试、接口测试和核心流程回归测试。影响范围较大的版本，需要测试负责人出具测试报告。

严重缺陷未关闭时不得上线。若业务方要求带风险上线，必须由产品负责人、研发负责人和业务负责人共同确认。

## 4. 发布流程

生产环境发布窗口原则上安排在工作日 20:00 到 22:00。紧急修复可以临时发布，但必须说明原因并保留操作记录。

发布前需要确认数据库脚本、配置变更、依赖服务、监控指标和回滚方案。发布后 30 分钟内，研发和运维应共同观察错误率、响应时间、CPU、内存和关键业务指标。

如果发布后出现严重异常，应在 10 分钟内决定是否回滚。回滚完成后需要补充故障复盘。

## 5. 故障分级

P1 故障指核心业务完全不可用、重大数据错误或大量客户受影响。P1 故障应在 15 分钟内建立应急群，30 分钟内给出初步处理方案。

P2 故障指部分功能不可用、少量客户受影响或性能明显下降。P2 故障应在 1 小时内给出处理方案。

P3 故障指非核心功能异常、展示问题或低频问题。P3 故障应在 2 个工作日内完成修复计划。

所有 P1 和 P2 故障都必须进行复盘，复盘内容包括时间线、根因、影响范围、处置过程和预防措施。
